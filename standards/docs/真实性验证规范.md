# 文创指南 - 真实性验证与交叉验证规范

## 核心原则

> **坚决与AI幻觉作斗争**
> 
> 所有数据必须经过真实性验证，拒绝虚构、拒绝推测、拒绝无法验证的信息。

---

## 1. 验证层级体系

### 三级验证机制

```
┌─────────────────────────────────────────────────────────┐
│  Level 1: 来源验证 (Source Verification)                │
│  - 验证信息来源是否可靠                                  │
│  - 标记数据来源级别 (A/B/C/D)                           │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Level 2: 交叉验证 (Cross Verification)                 │
│  - 至少2个独立来源交叉确认                               │
│  - 不一致时标记待核实                                    │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Level 3: 有效性验证 (Validity Verification)            │
│  - 验证链接可访问                                        │
│  - 验证图片可加载                                        │
│  - 验证数据时效性                                        │
└─────────────────────────────────────────────────────────┘
```

## 2. 品牌案例验证标准

### 2.1 来源验证清单

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| 品牌存在性 | 企查查/天眼查 | 有工商注册信息 |
| 官网验证 | 访问官网 | HTTP 200，内容匹配 |
| 电商店铺 | 天猫/京东搜索 | 有官方旗舰店 |
| 媒体报道 | 百度/谷歌搜索 | 有主流媒体报道 |

### 2.2 交叉验证要求

**必须提供至少2个独立来源：**

```
✅ 合格示例:
   来源1: 故宫淘宝天猫旗舰店 (https://gugong.tmall.com)
   来源2: 故宫博物院官网文创板块 (https://www.dpm.org.cn)

❌ 不合格示例:
   来源1: 某小红书笔记
   来源2: 另一篇小红书笔记 (同一平台，不算独立来源)
```

### 2.3 关键字段验证

| 字段 | 验证方式 | 验证工具 |
|------|----------|----------|
| 品牌名称 | 官网/电商店铺核对 | 浏览器访问 |
| 所在地区 | 工商注册信息 | 企查查API |
| 产品信息 | 电商店铺核实 | 爬虫抓取 |
| 销售额 | 财报/媒体报道 | 财经网站 |

## 3. 产品案例验证标准

### 3.1 图片验证

```python
# 图片有效性验证
验证项:
  1. URL可访问 (HTTP 200)
  2. 图片格式正确 (jpg/png/webp)
  3. 图片大小合理 (>10KB, <10MB)
  4. 内容相关性 (非随机图片)
  
验证工具:
  - curl -I {image_url}
  - Python requests + PIL
  - 人工抽检
```

### 3.2 价格验证

| 验证方式 | 说明 | 可信度 |
|----------|------|--------|
| 电商实时抓取 | 从天猫/京东实时获取 | A级 |
| 官网标价 | 品牌官方商城 | A级 |
| 媒体报道 | 主流媒体报道的价格 | B级 |
| 历史数据 | 过往收集的数据 | C级 |
| 估算 | 基于同类产品价格估算 | D级 |

### 3.3 销量验证

**销量数据来源可信度排序：**
1. 品牌财报/招股书 (最可信)
2. 电商平台公开数据
3. 行业报告引用
4. 媒体报道
5. 社交媒体讨论热度 (仅供参考)

## 4. 政策文件验证标准

### 4.1 政策真实性验证

**必须验证的要素：**

```
✅ 文号格式验证
   - 国务院: 国发〔年份〕序号
   - 部委: XX发〔年份〕序号
   - 地方: X政发〔年份〕序号

✅ 发布机构验证
   - 官网域名验证
   - 公章/落款核对

✅ 发布时间验证
   - 在合理时间范围内
   - 与文号年份一致

✅ 内容完整性验证
   - 有完整的政策文本
   - 有明确的适用范围
   - 有具体的政策措施
```

### 4.2 政策来源要求

| 优先级 | 来源 | 验证方式 |
|--------|------|----------|
| P0 | 政府官网 | 直接访问.gov.cn域名 |
| P1 | 官方公报 | 核对公报期号和页码 |
| P2 | 权威媒体 | 新华社、人民日报等 |
| P3 | 行业媒体 | 需二次验证 |

## 5. 验证工具与脚本

### 5.1 自动化验证脚本

```python
# verify_data.py - 数据验证脚本

class DataValidator:
    """数据验证器"""
    
    def verify_url(self, url):
        """验证URL可访问性"""
        try:
            resp = requests.head(url, timeout=10, allow_redirects=True)
            return resp.status_code == 200
        except:
            return False
    
    def verify_image(self, image_url):
        """验证图片有效性"""
        try:
            resp = requests.get(image_url, timeout=10)
            if resp.status_code != 200:
                return False, "无法访问"
            
            # 检查图片格式
            content_type = resp.headers.get('content-type', '')
            if not content_type.startswith('image/'):
                return False, "非图片格式"
            
            # 检查图片大小
            size = len(resp.content)
            if size < 1024:  # 小于1KB
                return False, "图片过小"
            if size > 10 * 1024 * 1024:  # 大于10MB
                return False, "图片过大"
            
            return True, "验证通过"
        except Exception as e:
            return False, str(e)
    
    def cross_verify_brand(self, brand_name, sources):
        """品牌交叉验证"""
        verified_count = 0
        for source in sources:
            if self.verify_url(source['url']):
                verified_count += 1
        
        return {
            'verified': verified_count >= 2,
            'source_count': len(sources),
            'verified_count': verified_count
        }
```

## 6. 数据标记规范

### 6.1 可信度标记

```json
{
  "verification": {
    "level": "A",
    "sources": [
      {
        "name": "天猫旗舰店",
        "url": "https://xxx.tmall.com",
        "verified": true,
        "date": "2026-02-21"
      }
    ],
    "cross_verified": true,
    "last_checked": "2026-02-21T12:00:00Z"
  }
}
```

### 6.2 验证状态标记

| 标记 | 含义 | 说明 |
|------|------|------|
| ✅ VERIFIED | 已验证 | 通过三级验证 |
| 🟡 PENDING | 待验证 | 已收集，待验证 |
| 🔴 UNVERIFIED | 未验证 | 无法验证，待删除 |
| ⚠️ PARTIAL | 部分验证 | 部分字段无法验证 |

## 7. 定期验证机制

### 7.1 验证周期

| 数据类型 | 验证频率 | 验证内容 |
|----------|----------|----------|
| 品牌案例 | 每季度 | 链接有效性、数据更新 |
| 产品案例 | 每月 | 价格更新、图片有效性 |
| 政策文件 | 每半年 | 政策时效性、链接有效性 |

### 7.2 失效数据处理

```
发现失效数据 → 标记为待核实 → 7天内验证 → 
  ├─ 验证通过 → 更新数据
  └─ 验证失败 → 删除数据
```

## 8. AI幻觉防范 checklist

### 数据收集阶段
- [ ] 不编造任何数据
- [ ] 所有数据必须有来源
- [ ] 不确定的数据标记为待验证
- [ ] 不使用"可能"、"大概"等模糊表述

### 数据处理阶段
- [ ] 不推测缺失数据
- [ ] 不合并不同来源的数据
- [ ] 保留原始数据备份
- [ ] 标注数据修改历史

### 数据发布阶段
- [ ] 所有数据经过验证
- [ ] 数据来源清晰标注
- [ ] 可信度等级明确
- [ ] 定期复查机制建立

---

## 9. 违规处理

### 发现AI幻觉数据
1. **立即下架** - 删除未经验证的数据
2. **追溯来源** - 查明数据产生环节
3. **流程改进** - 完善验证机制
4. **记录存档** - 记录问题，避免重复

### 数据质量事故分级
| 级别 | 定义 | 处理 |
|------|------|------|
| 轻微 | 个别字段不准确 | 修正并记录 |
| 中等 | 关键数据错误 | 删除并公告 |
| 严重 | 虚构数据/虚假信息 | 全面排查 |
